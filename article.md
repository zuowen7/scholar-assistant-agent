# 我用 Vibe Coding 耗时 12 天，写了一个纯本地 Rust 论文翻译引擎

现在全网充斥着"10 分钟写个网页"的 AI 编程演示。视频里 AI 噼里啪啦一顿输出，跑起来确实能看，但这种"开环控制"生成的产物，大多是缺乏生产环境鲁棒性的"工程玩具"：环境变量耦合、缺乏异常处理、逻辑边界模糊。

我花了 12 天，在 Claude Code 的协助下，通过 35 次严谨的 Commit 构建了一个全栈开源项目——**Scholar Translate**。

它不是一个简单的 API 套壳工具。它是一个基于 **Tauri (Rust) + Python FastAPI + 本地 Ollama** 架构，具备 16 种格式解析、双栏排版识别、字符级坐标推断能力，且完全物理隔离的桌面应用。

今天这篇文章，我不想聊 AI 有多神奇，我想聊聊我是如何通过标准软件开发生命周期（SDLC），去约束 AI 的随机性，从而交付一个工业级产品的。

## 一、顶层设计：为什么死磕"纯本地"与"双轨架构"？

在学术研究领域，数据隐私是生死线。未发表的论文、包含核心实验数据的课题报告，绝不能离开用户的控制域。因此，"断网可用"是本项目的核心逻辑。

在确立技术路线时，我制定了严谨的双轨架构：

> - **控制层 (Rust/Tauri)**：利用 Rust 的内存安全性，管理 Python 后端与 Ollama 推理引擎的完整生命周期
> - **逻辑层 (Python/FastAPI)**：利用 Python 成熟的 AI 与文档处理生态，执行复杂的 PDF 提取算法

这种架构面临的工程挑战远超单层应用。我们需要在 Rust 层实现底层的 TCP 端口探活（避开 WebView 网络栈的 CSP 限制），并利用 PyInstaller 将完整的 Python 运行时封装进 41MB 的独立安装包中。

这种从源码级到二进制分发级的控制，是"套壳脚本"无法企及的工程深度。

## 二、硬核解析：从空间分布统计到字符级推断

学术 PDF 的解析是公认的"脏活"。为了达到生产级的排版还原，我没有直接套用开源库的默认输出，而是设计了一套**五步流水线（Parser Pipeline）**。

在处理双栏排版时，我引入了**空间分布统计**：通过扫描页面中线的文字密度，动态判定布局——如果两侧均超过 50 个词且中心区域稀疏，即判定为双栏结构。

最核心的突破在于**词间距丢失的修复逻辑**：当 PDF 导出导致空格被吞噬时，系统会回退到字符级坐标分析。我通过计算相邻字符 Bounding Box（边界框）的间距，并建立一个基于当前字号的**动态阈值判定**。当间距超过字号的 20%~30% 时，判定为词间断层，强制注入空格占位符。

这种基于几何特征的启发式修复，实测能够应对 Science 等期刊的双栏排版。

在翻译环节，我设计了**三层上下文注入机制**：

> - **第一层**：系统提示词 + 自动维护的术语表（从已翻译文本中提取"中文（英文）"模式，保留最近 30 个高频术语对）
> - **第二层**：文档级上下文（标题 + 摘要），让模型理解整篇文章的研究领域
> - **第三层**：滑动窗口（前 800 字符已翻译文本），保证前后段落的语义连贯

翻译完成后还有 8 步后处理链：去除模型的思考标签、清理代码块包裹、检测行级和句级的循环重复、修复截断词……这些都是在实际翻译中遇到的异常输出，逐步积累下来的防御规则。

## 三、工程底色：用严谨的 SDLC 约束 AI 的随机性

这是本项目能够从"代码堆砌"升华为"工程产品"的关键。AI 只是极佳的执行元件，而整个**软件开发生命周期（SDLC）**必须由人类架构师掌控。

**1. 标准化流程与架构定义**

在敲下第一行代码前，我先制定了项目的 SOP。我明确定义了各模块间的 API 契约和数据流向，将 AI 的活动范围限制在原子化的功能函数内。架构设计是人定的，AI 的任务只是在边界内进行高效率的填充。

**2. 模块化开发与单元测试**

大模型的代码生成具有不确定性。为了对冲风险，我要求每个核心模块配备独立的单元测试。整个项目拥有 792 行专门的测试代码，覆盖了解析、清洗、切块、翻译、格式化五个管道阶段。只有通过了边界条件覆盖测试的代码，才允许进入主分支。

**3. 集成测试与阶段式流水线**

面对多进程交互这种高耦合任务，我将集成过程拆分为四个受控阶段：

> - **Phase 1**：漏洞审查——AI 自动审查出 13 个路径遍历与异常处理漏洞并修复
> - **Phase 2**：质量调优——并发控制与指数退避重试机制
> - **Phase 3**：生命周期绑定——健康检查与进程保活
> - **Phase 4**：代码规范化——防御性编程与日志埋点

每个阶段完成后展示交付物清单，**我确认 proceed 后才进入下一阶段**。

**4. 先验证，再修复**

大模型有一个致命问题——**幻觉 Bug**。它会非常自信地告诉你"这里有一个路径遍历漏洞"，但如果你不验证直接让它改，你会发现它声称的 Bug 有时根本不存在。

我的硬性规定是：**审查报告中提到的每一个 Bug，必须先读取对应代码确认问题确实存在，才能动手修改。** 这条规则帮我避免了至少 3 次基于错误假设的修改。

## 四、总结：基底信号决定 AI 上限

12 天，35 次提交，9000+ 行代码，覆盖了 Python 后端、Vue 前端、Rust 桌面端、Docker 部署及 GitHub Actions CI/CD。

很多人问我：Vibe Coding 是不是意味着程序员不再需要底层功底了？

恰恰相反。AI 是一个极强的放大器，但它放大的前提是你本身有高质量的"基底信号"。如果你不懂 SDLC 流程，不懂如何设计模块间的解耦，不懂如何用单元测试去界定逻辑边界，那么 AI 只会带你飞快地冲向工程崩溃。

真正构成技术壁垒的，是你对系统架构的理解力、对算法边界的判断力，以及对工程纪律的敬畏。
