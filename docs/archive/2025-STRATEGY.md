# Scholar Assistant 战略方向与团队分工

## 一、ChatGPT 诊断的准确性评估

逐条对照代码与事实：

| ChatGPT 指出 | 实际情况 | 评估 |
|---|---|---|
| 术语一致性弱，缺用户确认机制 | Glossary 已实现自动提取 + 跨 chunk 沿用，但无用户锚点锁定 | ✅ **属实**，应补交互层 |
| 长难句需改进拆分 | Chunker 只有 sentence/paragraph/fixed 三种基本策略，无从句感知拆分 | ✅ **属实** |
| 排版兼容差，图表公式不保位置 | 当前纯文本提取 + Markdown 输出，确实无视觉坐标映射 | ✅ **属实** |
| 对比阅读模式不足 | App.vue 已有双语对照 viewer，但不如 Scholaread 段落级重排 | ⚠️ 半属实，已有基础但不够精 |
| 性能串行、可并行 | Translator 当前逐块串行调用，确可并行优化 | ✅ **属实** |
| 缺少 OCR | Parser 不支持扫描件 OCR | ✅ **属实** |
| 不支持多语言 | 当前仅中英 | ✅ **属实** |

### 遗漏的关键事实
- ChatGPT **完全没提到**项目已有的 Agent 子系统（ReAct 循环 + RAG + 工具调用）
- **完全没提**学术写作 AI 功能（润色/扩写/连贯性/合规检查）
- **完全没提**LaTeX 模板导出 + Pandoc 编译
- **完全没提**双翻译引擎架构（本地 Ollama + 17 个云端 provider）
- 误以为 PDF 是唯一格式，实际支持 16 种

## 二、产品方向定位

### 一句话定位
> **隐私优先的学术 AI 写作辅助平台 — 从翻译切入，覆盖阅读、写作、排版全流程。定位在「DeepL 学术版」与「本地版 NotebookLM」之间。**

### 核心差异化（与竞品对比）

```
                通用翻译           学术阅读         PDFMathTranslate    我们
              (DeepL/Google)   (Scholaread/知云)   (29.8k stars)    Scholar Assistant
隐私/离线         ❌                  ✅ 部分           ✅ 本地可选         ✅ 全程本地
翻译质量          ❌ 通用语境         ✅ 学术术语库      ✅ DocLayout-YOLO   ✅ Glossary+上下文
格式保留          ❌ Markdown only   ✅ 原文重排        ✅ 原格式叠加        ⚠️ 文本提取(P1)
文献管理          ❌                 ✅ Zotero 集成     ✅ Zotero 插件      ❌
AI 写作辅助       ❌                 ❌                ❌                 ✅ Agent+RAG+Skill
LaTeX 导出       ❌                 ❌                ❌                 ✅ 6 套模板
MCP 支持         ❌                 ❌                ✅                  ❌ (P0 待做)
API 灵活度       ✅ 成熟 SDK        ❌                ❌                 ✅ 双引擎+多部署
跨页语义一致      ❌                 ❌                ❌                 ⚠️ (可从BabelDOC学习)
```

### 主要竞品分析

#### PDFMathTranslate（29.8k stars，EMNLP 2025）
- **优势**：DocLayout-YOLO 布局检测 + 原格式 PDF 叠加翻译、MCP Server 支持、Zotero 插件、Docker 部署、52+ 贡献者
- **劣势**：无 AI 写作辅助、无 Agent 对话、无 Skill 系统、无 LaTeX 模板导出、翻译引擎单一
- **我们不可竞争的领域**：PDF 格式保留（需要大量 CV/OCR 工作，非 2 人团队资源）
- **我们应该竞争的领域**：AI 写作辅助闭环、Agent 智能对话、学术工作流整合

#### BabelDOC（Immersive Translate 引擎）
- **优势**：跨页/跨栏语义一致性（解决了学术论文双栏排版的翻译断句问题）
- **劣势**：闭源、绑定 Immersive Translate 商业产品
- **可借鉴**：跨页断句恢复算法值得学习，可集成到 Chunker 的 sentence 策略中

#### 关键战略判断
> **我们不与 PDFMathTranslate 在「格式保留」上正面竞争。** 它是 52 人的开源项目，投入了 DocLayout-YOLO 等重量级 CV 模型。我们的差异化在「Agent + 写作辅助」：PDFMathTranslate 做的是"翻译后的 PDF"，我们做的是"翻译 + 理解 + 写作 + 排版的学术工作流"。

### 目标用户分层

| 用户画像 | 场景 | 付费意愿 |
|----------|------|----------|
| 硕博研究生（非 CS） | 读英文文献写中文论文 | 低（免费习惯强） |
| CS/AI 研究者 | 读论文、写英文论文、润色 | 中 |
| 涉密科研机构 | 离线翻译敏感文档 | 高 |
| 海外华人学者 | 中英双语写作 | 中 |

### 产品愿景：不做「大而全」，做「小而精的学术翻译+写作闭环」

**当前核心价值 = 翻译。** 以此为获客入口，向上生长 AI 写作辅助。

**不做的**（至少现阶段）：
- 不做文献管理（不跟 Zotero 竞争）
- 不做 PDF 阅读器（不跟 Scholaread 竞争）
- 不做通用翻译（不跟 DeepL 竞争）
- 不做 PDF 格式级排版保留（不跟 PDFMathTranslate 竞争 DocLayout-YOLO）
- 不做跨页语义重排（BabelDOC 的核心能力，需要大量 NLP 投入）

## 三、技术路线的排兵布阵

按「核心体验 → 差异化功能 → 商业化能力」三级划分：

### P0 — 核心翻译体验（当前：60% 完成度 → 目标：90%）

| 任务 | 说明 | 难度 | 工作量 |
|------|------|------|--------|
| **MCP Server 支持** | 将翻译管道、Agent 工具暴露为 MCP 协议，让 Claude/Cursor 等客户端直接调用（PDFMathTranslate 已支持，这是生态兼容的关键缺口） | 中 | 1-2 天 |
| **术语锚点系统** | 首次出现的术语弹出确认 → 全局锁定译法；用户可手动编辑术语表 | 中 | 2-3 天（后端+前端） |
| **并行翻译** | 多个 chunk 同时发请求（Ollama/Cloud），流式合并 | 低-中 | 1-2 天 |
| **扫描件 OCR** | 集成 Tesseract / PaddleOCR，作为 pdfplumber 的 fallback | 中 | 2-3 天 |
| **句法感知 Chunker** | 从句/括号/引语的边界感知拆分，不切断完整结构 | 中 | 2-3 天 |
| **双语对比阅读优化** | 段落级原文-译文并排滚动、点击定位、差异高亮 | 中 | 3-4 天（前端） |
| **状态持久化** | 翻译进度断点续传、结果缓存复用 | 低 | 1 天 |

### P1 — 学术写作差异化（当前：70% → 目标：95%）

| 任务 | 说明 | 难度 | 工作量 |
|------|------|------|--------|
| **Editor AI 面板打磨** | 润色/扩写结果用 diff 视图对比原文、一键应用/撤销 | 中 | 2-3 天 |
| **Agent 对话增强** | 支持对选中文本提问、引用上下文、追加约束 | 中 | 2-3 天 |
| **参考文献格式助手** | Agent 按 BibTeX 条目生成引用格式（GB/T 7714 / APA / IEEE） | 低 | 1 天 |
| **翻译结果导出 Word** | 用 python-docx 生成 .docx，保留段落结构 | 低 | 1 天 |

### P2 — 工程健壮性（当前：40% → 目标：85%）

| 任务 | 说明 | 难度 | 工作量 |
|------|------|------|--------|
| **Ollama 进程守护** | Tauri 主进程定时 health check，崩溃自动重启 | 低-中 | 1-2 天（Rust） |
| **异常处理体系** | 翻译失败的 Graceful Fallback（降级到云端 / 跳过坏块 / 提示用户定位） | 中 | 2 天 |
| **单元测试补全** | 当前 49 个 → 目标 100+，覆盖边界情况 | 低 | 持续 |
| **Tauri Windows 打包 CI** | GitHub Actions 自动化构建 + 发布 | 中 | 2-3 天 |
| **性能基准测试** | 记录每阶段耗时、token 消耗、Ollama 推理时间 | 低 | 1 天 |

### P3 — 商业化 / 社区（远期）

- 离线安装包一键分发
- 云端 API Key 管理 + 用量统计
- 用户自定义 prompt 模板
- 多语言扩展（日/德/法）

## 四、两人团队分工方案

### 分工原则
- **不按「前后端」切**（会导致一个人永远在等另一个人）
- **按「功能模块」切**，每个人 Own 完整的一条垂直链路

### 推荐方案：Owner 制

```

├─────────────────────────────────────────────────────────────┤
```

### 协作模式

- **每日站会**（5 分钟）：今日 1 个核心目标，是否 Blocked
- **API 契约先行**：前端需要的新端点，A 同学先定义 OpenAPI 文档/类型，B 同学 mock 并行开发
- **同一模块改动**：比如术语锚点系统，A 加后端 API（术语列表 CRUD + 注入翻译流程），B 加前端交互（弹窗 + 编辑面板），两人各自完成自己的层后联调

### 技术选型建议

| 方向 | 建议 |
|------|------|
| 并行翻译 | `asyncio` + `asyncio.Semaphore` 控制并发度，`httpx.AsyncClient` |
| 句法解析 | `spaCy` 或 `stanza`，仅对长句（>30 词）预解析再拆分 |
| OCR | `PaddleOCR`（中文论文效果好）或 `Tesseract` |
| Word 导出 | `python-docx` |
| 前端状态管理 | 当前用 composable 模式已够，不用 Pinia 增加复杂度 |
| 测试框架 | `pytest` + `pytest-asyncio` |
| CI | `GitHub Actions`：Windows 构建 Tauri + Linux 跑 Python 测试 |

## 五、为什么 ChatGPT 的建议需要「选择性采纳」

**应该采纳的**：
- 术语锚点系统 ✅（提升翻译质量的核心交互）
- 并行翻译 ✅（性能优化，收益明确）
- 扫描件 OCR ✅（补 PDF 场景短板）
- 双语阅读优化 ✅（用户体验提升）
- Word 导出 ✅（实用需求）

**应该拒绝的**：
- ❌「与 Zotero/Crossref 集成做文献管理」— 方向性偏离，2 人团队做不了
- ❌「多端同步」— 需要服务端，偏离离线定位
- ❌「13 种语言」— 过早，先做好中英
- ❌「GPU 加速」— Ollama 自身已支持 CUDA，不需要额外工作
- ❌「翻译质量评分」— 主观指标，价值不明确
- ❌「DocLayout-YOLO 级 PDF 排版保留」— PDFMathTranslate 52 人团队已做好，2 人团队不该正面竞争

**应该新增的**（基于竞品分析）：
- ✅ MCP Server 支持 — Claude/Cursor 等客户端可直接调用翻译和 Agent 工具（1-2 天）
- ✅ 双语 PDF 叠加优先级提升 — 格式保留是用户第一痛点，不求 DocLayout-YOLO 级完美，但求实用
- ✅ 跨页断句恢复 — 借鉴 BabelDOC 思路，在 Chunker 中处理双栏跨页断句

**保持现状的**：
- LaTeX 模板导出 — 已经足够好，小修即可
- Agent + RAG — 已经完成，打磨交互即可
- Cleaner 管线 — 非常成熟（789 行，17 阶段），极稳定，不需大改

## 六、我的补充建议（ChatGPT 没提到的方向）

### 1. 翻译记忆库

**问题**：同一段文字在不同文档反复出现时，每次都要重新调用 LLM，耗时且结果可能不一致。  
**方案**：利用已有的 ChromaDB 存储 `原文 embedding → 译文` 映射，翻译前查 TM，命中直接复用。  
**收益**：减少 10-30% API 调用，跨文档术语一致。  
**实现**：1-2 天，加 `TMCache` 类，翻译管线中 `TM → LLM → TM` 三步。

### 2. 文档级一致性质检

**问题**：当前每块独立翻译，最后没有全文维度的质量把关。  
**方案**：翻译完成后用轻量 prompt 扫描全文译文，标记术语/风格不一致。  
**收益**：从「每段尽力」到「全文可控」。  
**实现**：`Translator.post_process()` 加 `_consistency_check()`，返回警告列表前端展示。

### 3. 双语 PDF 叠加（最佳格式保留方案）

**问题**：纯文本输出丢失原始排版，图表公式位置全无。  
**方案**：Parser 输出时保留 `(page_no, bbox)` 坐标，翻译后用 PyMuPDF 在原 PDF 上绘制译文字块，输出「双语叠加 PDF」。  
**收益**：最务实的格式保留方案——不需要复杂排版引擎，用户在原 PDF 上直接看双语。  
**实现**：3-5 天，Parser 改造保留坐标，Formatter 新增 `pdf_overlay` 模式。

### 4. 多轮翻译精炼

**问题**：一次翻译定稿，首轮质量不足无改进手段。  
**方案**：第一轮「快速直译」（低 temp + 短 context），第二轮「学术润色」（用润色 prompt 改写译文 + 注入 Glossary）。  
**收益**：长难句翻译质量显著提升。  
**实现**：1-2 天，Translator 支持 `refine=True` 参数。

### 5. 用户纠错闭环

**问题**：翻译错了用户只能手动改，下次还错。  
**方案**：前端修改译文时，记录 `原文 → 用户修正` 对，存入 Glossary/TM，后续自动生效。  
**收益**：越用越准，形成用户专属翻译模型。  
**实现**：3-4 天，API 加修正记录端点，前端编辑时弹出「保存为术语?」。

### 6. Ollama 多模型路由

**问题**：所有翻译都用同一模型，短块用小模型足够。  
**方案**：短文本（标题/图注 < 100 字符）→ qwen3:4b（快速），长段落 → qwen3:32b（高质量）。  
**收益**：综合速度提升 2-3 倍。  
**实现**：1 天，Translator 加 `model_router(config)`。

### 7. api_factory.py 拆分为多文件

**问题**：单文件 ~900 行，路由+业务逻辑+进程管理揉在一起。  
**方案**：拆为 `routers/translate.py`、`routers/config.py`、`routers/agent.py`、`services/translation_pipeline.py`。  
**收益**：可维护性提升，单人开发不再互相冲突。  
**实现**：0.5 天，纯机械拆分，不改变接口。

### 8. Agent 改用 WebSocket

**问题**：SSE 单向推送，Agent 需要用户「中断/澄清/确认」时 SSE 无法双向。  
**方案**：Agent 对话改用 WebSocket，支持对话中途用户插话纠正方向。  
**收益**：Agent 交互体验从「单向流」升级为「真正对话」。  
**实现**：2-3 天，FastAPI WebSocket 端点 + 前端 WebSocket client。

## 七、风险与缓解

| MCP 协议生态未成熟 | 中 | 投入可能浪费 | MCP 已被 Claude/Cursor 支持，风险可控 |

## 九、开发原则与协作规范

### 代码约定

| 领域 | 约定 |
|------|------|
| Python | `ruff` 格式化 + 类型注解全覆盖，禁止 `Any`（除非调用外部库） |
| TypeScript | `vue-tsc` 严格模式，禁止 `any` |
| Rust | `cargo clippy` 无 warning |
| 提交信息 | `<type>(<scope>): <subject>` 格式，如 `feat(translator): add parallel chunk dispatch` |
| 分支策略 | 直接 push `main`（两人团队不需要 PR 流程），提交前两人各自确保 lint 通过 |

### 沟通规范

- **每日 5 分钟站会**（必做）：昨天做了什么 / 今天做什么 / 有没有 Blocked
- **不隔夜问题**：被 Blocked 超过 2 小时就找对方
- **做完一件事再切下一件**：避免两个人同时在改翻译管线

### 代码审查（简化版）

## 十三、写在最后：给两位开发者的建议

1. **宁愿少做一个功能，也不要做一个有 Bug 的功能**。学术翻译的用户对准确性极其敏感，一个术语翻译错误就足以让他们弃用。
2. **每一行代码都要有人负责**。不写「共享代码」——每段代码都明确属于 A 或 B 的 Owner 范围。共享意味着没人维护。
3. **每周用 1 小时读用户反馈**。看看 GitHub Issues 里用户在抱怨什么，比你自己拍脑袋决定优先级重要 100 倍。
4. **不要跟风加功能**。如果 DeepSeek 出了新模型、AI Agent 又有了新范式——只在它们解决你用户真实问题时才加。否则就是 feature creep。
5. **先让 10 个人爱上你的产品**，再想怎么让 1000 个人用。学术翻译是高频场景，用户愿意为质量付费。第一步是找到 10 个每周至少用 1 次的用户。
