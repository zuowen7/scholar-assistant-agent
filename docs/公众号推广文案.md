# 一个开源的学术 AI 工作台：翻译、写作、审稿、排版一条龙

写论文有个很烦的事：翻译 PDF 开个工具，理思路画个导图，改稿切到编辑器，检查逻辑漏洞全靠肉眼，最后排版还得折腾 LaTeX 模板。每个环节是通的，工具是断的。

Scholar Assistant（研墨）把它们收进一个桌面应用。今天开源，MIT 协议。

github.com/zuowen7/scholar-assistant

---

## 跟聊天框不一样

市面上 AI 写作工具几乎全是聊天界面——你问一句，它回一段。问题是论文不是聊天能写完的。

研墨的做法：你打开一个项目文件夹，它就是这个文件夹里的 Claude Code。Agent 有六个文件操作工具——`read_file`、`write_file`、`str_replace`、`grep_files`、`git_op`、`run_command`。你说「把 Introduction 第三段改凝练」，它自己定位、替换、保存。你说「编译 LaTeX 让我看效果」，它调 pandoc 编译。

访问工作区外的路径会弹审批窗，安全边界硬编码。

---

## 翻译：前处理比模型重要

调 API 翻译不难。难的是翻译之前。

PDF 解析出来是乱的。页眉页脚混进正文，换行符切断句子，跨页段落被硬拆开，单词里藏着连字符（hy-phenation）。这些问题 LLM 修不好——你给模型一个断掉的句子，它不知道上下文，翻译出来就是碎的。

所以写了 17 道清洗规则。去页眉页脚、修断行、合并跨页段落、还原连字符、识别引用区并跳过。

然后是 5 步流水线：parse → clean → chunk → translate → format。每一步状态 SSE 实时推到前端，不是黑箱请求。翻译完左右双栏对照，鼠标悬停句子级高亮对齐，像 DeepL 那样。

## Agent：手写的，没套壳

市面上做 Agent 的基本用 LangChain。我没用。

学术写作场景里每一步需要精确控制：翻译什么时候跳过引用区、失败块怎么重试、文档问答什么时候不该进多步循环。框架管不了这些判断。

自己写的代价是踩了不少坑。说两个：

第一，**21 家模型的 tool-calling 行为千差万别**。DeepSeek 在 assistant 消息里收到 null content 直接 400，但 OpenAI 经常返回 null。有的模型过度积极，问个定义也要先调搜索工具；有的消极怠工，该调工具却一直说「我需要查看文件」就是不发 tool_call。我只能给每个模型写对应的工具调用指导，嵌进 system prompt，才把行为统一。

第二，**Agent 死循环**。经典场景：用户打开文档问「这篇讲了什么」，Agent 读完后不确定要不要改，又读一遍，再读一遍。修法：纯文档问答直接走单次 LLM 回答，不进多步工具循环。只有明确要改文件或跑命令才启动完整 Agent。硬止损：单任务 20 步上限，全局 60 步，同一工具累计调用过多触发警告后强制终止。

## 论证账本：跟论文对账

这是最花时间的模块，也可能是最原创的。

让 LLM 读摘要和引言，拆出每一条「承诺」——「我们提出了 X 方法」、「相比基线提升 Y%」、「解决了 Z 问题」。然后去正文逐条验证：这条兑现了吗？证据在哪？证据够不够？

每条承诺和兑付点锚定到原文精确字符位置。你改稿后，系统用模糊匹配自动重定位，分三种状态：锚定、漂移、丢失。改完 Introduction 忘了后面说过的话？它会提醒你。

审稿模块模拟 7 种会议风格：NeurIPS、ICML、ICLR、ACL、CVPR、KDD、CHI。逐句评审，每条批评锚到源句。你可以在对话框里跟 reviewer 对线——有道理的话它真的会被说服。评审分三个角度（方法、实验、写作）用 `asyncio.gather` 并发跑，去重合并。

## 技术栈

| 层 | 技术 |
|---|---|
| 前端 | Vue 3 + TypeScript + Monaco Editor + Vue Flow |
| 桌面 | Tauri 2（Rust 管理子进程） |
| 后端 | Python FastAPI，全链路 SSE |
| 本地 LLM | Ollama + Qwen3:8b |
| 云端 LLM | OpenAI、Anthropic、DeepSeek、智谱、通义千问 等 21 家 |
| 向量检索 | ChromaDB + all-MiniLM-L6-v2 |
| 导出 | Pandoc + Tectonic，6 套 LaTeX 模板 |
